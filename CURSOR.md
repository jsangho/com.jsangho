# Cursor 하네스 지도

이 문서는 **Cursor에서 AI를 쓸 때** 이 저장소의 규칙·검증 루프를 어떻게 걸 것인지 정리한 **하네스 엔지니어링** 가이드다. 코딩 에이전트의 일반 원칙(가정을 드러냄, 최소 변경, 검증 가능한 목표)은 [Andrej Karpathy의 관찰](https://x.com/karpathy/status/2015883857489522876)을 한글로 풀어 쓴 `CLAUDE.md`를 따른다.

**트레이드오프:** 속도보다 신중함. 다만 “한 줄 고치기” 수준은 판단으로 가볍게 처리해도 된다.

---

## 하네스를 쪼갠 이유

| 층 | 파일 | 역할 |
|----|------|------|
| 레일 | `.cursorrules` | 매 세션에 자동으로 걸리는 **짧은** 행동 제약. **`docs/` 읽기 의무** 포함. |
| 업무 규칙 | 저장소 루트 **`docs/`** | Frontend/Backend 등 **스택별 코딩 규칙**. 구현 전에 반드시 Read. |
| 원칙서 | `CLAUDE.md` | 같은 철학의 **전체** 설명·예시·검증 문장. 모호할 때 여기서 기준을 잡는다. |
| IDE 지도 | `CURSOR.md`(본 문서) | Cursor 기능과 **어떻게** 맞출지(컨텍스트, 검증, 우선순위). |

에이전트는 위 네 층을 **한 세트의 하네스**로 본다. 스택별 패턴은 `.cursorrules`가 아니라 `docs/`에만 둔다.

## Karpathy 의도를 Cursor에서 실현하는 방법

### 1. 구현 전 사고 → 컨텍스트를 명시적으로 건다

- `@파일`, `@폴더`, `@Codebase`로 **근거가 될 코드**를 붙인다. 에이전트가 저장소 전체를 “안다”고 가정하지 않는다.
- 요구가 두 가지로 읽히면, 채팅에서 **해석 후보를 나열**하거나 되묻게 한다(에이전트도 `CLAUDE.md`에 따라 조용히 한쪽을 고르지 않는다).

### 2. 단순성·정밀 수정 → 변경 단위를 작게

- 한 턴에 “리팩토링 + 기능 + 문서”를 섞지 않는다. 티켓/질문을 **한 목적**으로 쪼갠다.
- 에이전트에게 “인접 파일 예쁘게 정리”를 시키지 않는다. 필요한 경로만 `@`로 제한하는 편이 낫다.

### 3. 목표 중심 실행 → 검증을 채팅 밖으로 빼지 않는다

- “돌아가게” 대신 **끝났다고 말할 수 있는 조건**을 같이 적는다. (예: `npm test` 통과, 특정 API 200, 재현 스텝 소멸)
- 에이전트에게 실행·검증을 맡길 수 있으면 터미널·도구로 돌리게 한다. 출력 없이 추측으로 마친다고 하지 않게 한다.

---

## 우선순위(충돌 시)

1. **사용자가 그 대화에서 명시한 지침**
2. **`docs/` 업무 규칙** (해당 스택의 `*_RULES.md`)
3. `.cursorrules`
4. `CLAUDE.md`의 일반 원칙  
5. (있다면) `.cursor/rules/` 아래 규칙·`AGENTS.md` 등 프로젝트 확장

명시가 없으면 위 순서로 **일관되게** 적용한다. Backend 작업 시 채팅에 `@docs/DevOps/Backend/BACKEND_RULES.md`를 붙이면 §0 절차를 빠르게 충족할 수 있다.
